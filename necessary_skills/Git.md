# Git

Please use
Git for managing this source code! If you do not know Git, LEARN IT! It is so
much more organized to keep and manage different versions of your code than
when you do this by making a bunch of different project folders all over the
place and creating archives (.zip files) of key stages of your work. If you
want to keep a copy of your code that represents stable code corresponding to a
journal publication for example, just make a Git branch and name it something
informative. Or better yet, simply tag the commit of interest with a name that
indicates it as the code corresponding to a journal publication. Then if anyone
ever needs to see exactly what was going on, they can just checkout that
commit. These tags or branches should be documented in this README. Since this
project relies on the Robotics Framework, you may consider keeping track of
the commit in the Robotics Framework repo that was in use for a particular tag.
Or you can include the hash of the commit in this README. Just make sure there
is enough information to recreate the codebase exactly as it was when the
publication was accepted. Furthermore, don't forget to push the repo changes
to the remote respository hosted on the Telerobotics Lab's Bitbucket account so
the repo is backed up on the cloud. This avoids the need to fill up the lab's
shared drive with many versions of source code.
